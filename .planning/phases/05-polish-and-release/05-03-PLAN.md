<task type="checkpoint:decision" gate="blocking">
  <name>Task 1: Decide isDesktopOnly value before release build</name>
  <action>Evaluate whether mobile Obsidian has been tested and set isDesktopOnly in manifest.json accordingly.</action>
  <decision>Set isDesktopOnly in manifest.json before the release build</decision>
  <context>
    The current manifest has isDesktopOnly: false. The plugin uses only requestUrl() (not Node.js fetch or fs), which is mobile-safe in theory. However, mobile Obsidian has not been tested. Per CONTEXT.md: if mobile cannot be tested before submission, flip isDesktopOnly: true for v1.

    Setting isDesktopOnly: true means mobile users cannot install the plugin until it is updated.
    Setting isDesktopOnly: false and submitting without mobile testing risks a Settings tab crash on mobile that could cause a reviewer to reject the submission.
  </context>
  <options>
    Option A - desktop-only (safe, recommended): Set isDesktopOnly: true. No risk of mobile crash rejection. Honest about what has been tested. Can flip to false in v1.1 after mobile verification.

    Option B - mobile-enabled (requires verification): Keep isDesktopOnly: false. Must verify on physical mobile Obsidian device first: enable plugin, open Settings tab, confirm no crash. If device is unavailable, choose Option A.
  </options>
  <verify>Manual - human decision required</verify>
  <done>User selects Option A or Option B (with confirmation that mobile was tested if B).</done>
  <resume-signal>Select: desktop-only OR mobile-enabled (and describe what you tested on mobile if choosing mobile-enabled)</resume-signal>
